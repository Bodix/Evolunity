# 🎲 Loot Table: Руководство и Ограничения

Добро пожаловать в документацию системы генерации лута. Наша таблица лута построена на архитектурном паттерне **Компоновщик (Composite)**. Это означает, что таблица представляет собой последовательный список независимых "Генераторов" (узлов).

При генерации система идет строго **сверху вниз**, проверяя каждый узел по очереди. Узлы ничего не знают друг о друге. Это делает систему невероятно надежной и предсказуемой, но накладывает определенные ограничения на сложную логику.

Ниже описаны механики, которые **НЕВОЗМОЖНО** реализовать базовыми средствами этой таблицы, и то, как с этим работать.

---

## 🎨 Часть 1. Для Геймдизайнеров: Ограничения логики

Так как наша таблица является **Stateless** (не запоминает, что происходило на предыдущих шагах генерации), вы столкнетесь со следующими ограничениями при балансировке:

### ❌ 1. Глобальные лимиты (Global Drop Caps)

* **Чего нельзя сделать:** Сказать таблице: *"Пройдись по всем 10 узлам, но как только выпадет 3 любых предмета — немедленно остановись!"*.
* **Почему:** Таблица честно отработает все узлы. Если сработают все 10 вероятностей, выпадет 10 предметов.
* **Как решать:** Таблица лута не должна заниматься такими ограничениями. Этим должен управлять внешний скрипт (например, `ChestManager`). Таблица может выдать 5 предметов, а скрипт сундука сам выберет из них 3 случайных и отдаст игроку.

### ❌ 2. Динамическое исключение из пула (Deck-building effect)

* **Чего нельзя сделать:** Внутри одного взвешенного пула (`WeightedPoolDrop`) поставить 3 броска кубика (`Rolls = 3`) и сказать: *"Если первым броском выпал Уникальный Меч, убери его из пула, чтобы оставшиеся 2 броска не могли выдать его дубликат"*.
* **Почему:** Веса внутри пула статичны для всех бросков в рамках одной генерации. Меч может выпасть трижды.
* **Как решать:** Если предмет строго уникален (падает только 1 раз), его нужно выносить в отдельный узел `ItemDrop` со своей вероятностью, а не класть в общий пул с множественными бросками.

### ❌ 3. Последовательные зависимости (Sequential Dependencies)

* **Чего нельзя сделать:** *"С шансом 50% падает случайное Оружие. Если оружие выпало, то следующим предметом ОБЯЗАТЕЛЬНО должны выпасть патроны именно для этого оружия"*.
* **Почему:** Узел, генерирующий патроны, не знает, что сгенерировал предыдущий узел с оружием.
* **Как решать:** Использовать "Наборы" (Bundles). Вы создаете отдельный ScriptableObject `LootBundle`, который содержит и Оружие, и Патроны, и кладете в таблицу именно его как единый предмет.

### ❌ 4. Перемешивание правил (Randomized Node Order)

* **Чего нельзя сделать:** Заставить таблицу оценивать узлы в случайном порядке.
* **Почему:** Список всегда выполняется строго сверху вниз. В большинстве случаев порядок не имеет значения, но важно помнить об этом при проектировании шансов.

## 💻 Часть 2. Для Разработчиков: Технические нюансы

Под капотом система использует атрибут `[SerializeReference]` для хранения полиморфного списка генераторов в одном массиве. А для совместимости с **Unity 2019.4** мы используем не-generic базовые классы. Это накладывает строгие технические правила.

### ⚠️ 1. Хрупкость рефакторинга (Refactoring Fragility)

Это самое критичное ограничение движка Unity.

* **Проблема:** Если геймдизайнеры уже заполнили таблицы лута в проекте, а вы решите переименовать класс (например, `ItemDrop` в `SingleItemDrop`) или изменить его `namespace`, **Unity безвозвратно потеряет все данные этих узлов**. В инспекторе появится ошибка *Missing Type*.
* **Решение:** Если вы меняете имя класса или пространство имен, вы **ОБЯЗАНЫ** использовать атрибут `[MovedFrom]` из `UnityEngine.Scripting.APIUpdating`.

```csharp
[MovedFrom(true, "OldNamespace", "OldAssembly", "OldClassName")]
[Serializable]
public class NewClassName : LootDrop { ... }

```

### ⚠️ 2. Ограничения стандартного UI (Inspector UX)

Пока не написан полноценный Custom Editor, стандартный инспектор Unity имеет проблемы с `[SerializeReference]`:

* **Copy / Paste:** Не пытайтесь копировать элементы полиморфного списка через контекстное меню (Copy/Paste Array Element). Unity может скопировать их по ссылке, что приведет к изменению данных в обоих узлах одновременно.
* **Undo / Redo:** Нажатие `Ctrl+Z` при добавлении или удалении полиморфных узлов в массиве иногда может вести себя непредсказуемо в старых версиях Unity.

### ⚠️ 3. Boilerplate-код для новых типов таблиц (Из-за Unity 2019.4)

Из-за ограничений сериализатора Unity 2019.4, который не поддерживает отображение обобщенных полей (generic fields), мы не можем использовать `ItemDrop<T>` напрямую в инспекторе.

* **Правило:** Для каждого нового типа генерируемого объекта (например, `ResourceData`) вам придется создавать классы-обертки для сериализации:

```csharp
// Это необходимо только для того, чтобы Unity 2019.4 увидела поля в инспекторе.
[Serializable] public class ResourceItemDrop : ItemDrop<ResourceData> { }
[Serializable] public class ResourceWeightedEntry : WeightedPoolEntry<ResourceData> { }
[Serializable] public class ResourceWeightedPoolDrop : WeightedPoolDrop<ResourceData, ResourceWeightedEntry> { }

```

## ✅ Что система делает идеально?

Несмотря на ограничения, текущая архитектура блестяще справляется с:

1. **Смешиванием логики:** Вы можете в одной таблице иметь и 100% гарантированные дропы, и независимые шансы, и классические "гача"-пулы с весами.
2. **Пустыми бросками (Empty Rolls):** Поддержка шанса вытащить "ничего" из взвешенного пула.
3. **Безопасностью (Fail-fast):** Любые кривые настройки от геймдизайнера (отрицательные веса, min > max) блокируют только сломанный узел, логируют понятную ошибку и не крашат весь поток выполнения игры.
